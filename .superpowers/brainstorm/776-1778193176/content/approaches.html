3 enfoques arquitectónicos

Elige el que mejor encaja con tus objetivos. Hay una recomendación clara.

A

Mono-app · Route Groups · Drizzle ★ Recomendado

Una sola Next.js app con dos route groups: (customer) para el flujo QR y (admin) para el panel del restaurante. Drizzle ORM sobre Neon. Server Actions para todo. PDF generado en servidor con @react-pdf/renderer. Auth solo en el grupo admin.

Pros

  • Un solo repo, un solo deploy
  • Drizzle: TypeScript-first, migrations, zero overhead
  • Server Actions = integración POS futura trivial
  • FSD encaja naturalmente en route groups

Contras

  • PDF pesado bloquea el servidor si hay volumen alto (mitigable con streaming)
B

Monorepo Turborepo · apps/customer + apps/admin

Dos Next.js apps independientes en un monorepo pnpm/Turborepo. Paquetes compartidos: @repo/db (Drizzle+Neon), @repo/validators (Zod), @repo/pdf. Deployments separados en Vercel.

Pros

  • Separación total: el panel admin no carga nada del flujo cliente
  • Escala y deploya de forma independiente

Contras

  • Setup inicial más costoso
  • Overkill para un restaurante o cadena pequeña
  • Dos deploys = más superficie de fallo
C

Next.js · API Routes como backend explícito

En vez de Server Actions, toda la lógica de datos pasa por Route Handlers REST (/api/invoices, /api/sessions). El cliente llama a la API con fetch. Más próximo a una arquitectura API-first.

Pros

  • API reutilizable si el POS necesita llamarla directamente
  • Más fácil de testear con herramientas HTTP

Contras

  • Más boilerplate que Server Actions
  • Pierde co-locación tipo + validación que da Zod+Actions
  • Menos FSD-friendly: la lógica de negocio se dispersa
Mi recomendación: A — Una sola app bien estructurada con FSD es más que suficiente para este dominio. Drizzle + Server Actions dan type-safety end-to-end sin monorepo overhead. Si el volumen crece, la migración a Turborepo es incremental.
