Sección 4 — Errores, edge cases y testing

Casos límite del flujo cliente y estrategia de tests

Edge cases críticos
Mesa sin sesión activa
RSC → página de error amigable: "No hay ningún consumo abierto en esta mesa." Sin stack trace expuesto.
Sesión ya facturada
Server Action detecta estado = 'facturada' → error 409. Mensaje: "Ya se emitió una factura para esta mesa." Evita duplicados.
Doble envío concurrente
UNIQUE constraint en facturas(sesion_id). El segundo insert falla a nivel DB → el primero gana, el segundo recibe error 409.
Fallo en generación PDF
La factura ya está en DB. pdf_url queda null. Página de confirmación muestra botón "Regenerar PDF" que llama a otra Server Action.
Fallo en envío email
No-op silencioso. La factura está guardada, el PDF disponible. El panel admin muestra un badge "Email no enviado" si Resend reporta error.
Estrategia de testing
Unit — Vitest
shared/lib/validators/spanish-id.ts → suite exhaustiva con casos válidos/inválidos para NIF, CIF y NIE (Módulo 11).
shared/lib/tax/iva.ts → cálculo de bases imponibles y cuotas IVA.
Integration — Vitest + Neon branch
Server Actions testeadas contra una branch de Neon de test. Cubre: emitInvoice() happy path, duplicados, sesión ya facturada.
E2E — Playwright
Golden path: escanear QR → rellenar formulario → descargar PDF.
Edge: mesa sin sesión, NIF inválido.
Sin mocks de DB
Los tests de integración usan Neon branching — cada run de CI crea una branch efímera, aplica migrations y la destruye al terminar.
